# QQ 产品经理 PRD 工作流 (QQ PRD Workflow)

# 触发方式

当用户提供产品初步想法或需求描述时，请严格按照以下**七个步骤**逐步引导，确保流程规范、需求落地无偏差。

**🚨 绝对强制指令：必须分步执行（Step-by-Step），严禁一次性输出所有步骤结果！所有带有【等待用户确认】的步骤结束后，需立即停止输出，等待用户反馈后再推进下一步。**

**🎯 PRF初始表格：可选引入 prf.md （Product Requirements Form）指定项目名称、标识，AI会根据表格内容初始化项目需求。**

**🎨 模板强制使用指令：在生成任何文档时，必须严格使用 `/template` 文件夹下对应的模板文件样式和结构，确保输出格式统一、专业。**

**🧑‍🎨 样式强制使用指令：在生成html格式文档时，必须严格使用 `/assets/vercel/DESIGN.md` 样式定义文件，确保输出格式统一。**

## 步骤一：对话式需求采集与确认

新手产品经理往往难以一次性清晰梳理需求，AI需采用「灵活对话式」引导方式，主动评估需求完整性、温和追问补充，完成需求确认后再推进后续工作。核心原则：**需求未明确前，绝不急于创建文件或文件夹**。

### 1.1 接收需求

- 兼容用户任意形式的需求输入（一句话描述、截图、手绘图、口语化表述等），无需用户调整格式。

- 接受用户提供 **Product Requirements Form** 需求表格，用于初始化项目，结构模板见 `/references/prf_template.md`

- 若用户没有提供需求表格，不要求用户提供，按以后的流程进行对话式需求评估。

### 1.2 核心维度评估与反馈

接收用户初步需求后，**必须将以下7个关键维度的评估结果清晰呈现给用户**，明确告知当前需求的清晰项与缺失项，引导用户补充完善：

1. **背景/痛点**：需求提出的初衷是什么？当前存在哪些未解决的问题？（必选，核心前提）

2. **业务目标**：需求落地后，期望达成的具体业务效果是什么？（必选，核心导向）

3. **用户与场景**：需求针对的核心用户是谁？用户将在什么具体场景下使用该功能？（必选，落地基础）

4. **核心用户旅程**：用户从接触功能到完成核心目标的关键路径是什么？路径中存在哪些触点和潜在痛点？（必选，流程核心）

5. **产品范围**：明确需求属于新产品立项，还是现有产品的迭代升级？（必选，范围界定）

6. **现有方案**：目前用户是通过什么方式解决该问题的？存在哪些必须遵守的业务限制或特定流程？（重要，规避冲突）

7. **参考/竞品**：是否有喜欢的对标产品或参考案例？（加分项，优化方向）

### 1.3 深度追问与多轮确认 【等待用户确认】

**🚨 关键规则：不急于总结需求，必须完成至少3轮深度启发式追问，确保需求无遗漏、无歧义！**

- **追问方式**：摒弃死板的选项式提问（A/B/C），采用**开放式、启发式**提问，引导用户全面思考需求细节。

- **追问数量**：每轮追问需包含 **3-5个问题**，聚焦需求盲区，逐步挖掘细节。

- **循环机制**：基于用户上一轮的回答，针对性挖掘新的需求盲点，持续追问，至少完成3轮对话。

- **最终确认**：仅当通过多轮追问完全理解需求所有细节，且用户明确表示“没有补充了，可以开始写了”，方可对需求进行结构化复述，进入下一步流程。

- **🛑 强制暂停：每一轮提问结束后，必须停止输出，等待用户回复后再进行下一轮操作！**

## 步骤二：项目初始化与文件夹架构搭建

需求确认无误、准备启动文档产出前，需先为用户搭建清晰、规范的项目文件夹结构，避免后期文件混乱、查找不便，为后续版本管理和协同工作奠定基础。

### 2.1 建立标准文档文件夹树

**必须创建标准文档文件夹树** 创建**以英文项目名称+PRD命名的专属根文件夹**，如`/quizgame_prd/`、`/ecommerce_prd/`，方便后期分发打包，并在根文件夹内创建以下标准文件及子文件夹，严格遵循命名规范，并强制使用文件夹`./temlate/` 中的模板文件样式和结构，确保文档格式统一、专业：

- `index.html` ：主控框架文档，实现PRD版本切换、多产品形态原型切换、PRD与原型左右联动等核心功能。

- `v1.0/`：按版本分类存放当前版本所有相关文档（后续迭代版本依次创建v1.1/、v1.2/等）。

- `v1.0/prd_md/` ：存放**Markdown格式 PRD 源文档**，命名规范：`prd_v1.0.md`。

- `v1.0/prd_html/` ：存放**HTML格式PRD文档**（与**Markdown格式 PRD 源文档**同步），命名规范：`prd_v1.0.html`。

- `v1.0/prototype_html/`：存放所有高保真 **HTML格式 原型文件**，按产品形态分文件管理，命名规范：`admin_prototype_v1.0.html`（后台端/PC Web）、`app_prototype_v1.0.html`（移动端/Mobile Web），子界面文件使用文件夹管理，如`admin_pages/login.html`、`app_pages/register.html`。

- `assets/`：存放Mermaid流程图、导出图片、附件（Excel/CSV数据字典、需求原始文稿、参考资料等）、app端静态资源、后台静态资源等。

- `history/` ：存放项目修改历史日志，方便多Agent协同回溯，命名规范：大模型英文名称 + 日期 + 版本 + 摘要（例：`kimi_2026_03_27_v1.0_init.md`），同一版本按天按需多次更新，确保记录最后的修改状态，清理不重要的反复修改信息，确保简洁、完整、准确。

### 2.2 确认架构

向用户清晰展示搭建好的文件夹结构，明确告知后续所有产出物（PRD、原型、流程图等）将按类别存放在对应文件夹中，确保用户了解文件存放逻辑。

## 步骤三：输出"详细的第一版初步 PRD"

**🚨 关键规则：初步 PRD 需具备完整性和详细性，不可仅提供空框架；需同步生成 Markdown 格式 源文档和 HTML 格式浏览用文档，确保两者内容一致。**

### 3.1 产出内容

基于步骤一确认的需求信息，**严格参考 `/template/v1.0/prd_md/prd_v1.0.md` 标准模板结构和 `/template/v1.0/prd_html/prd_v1.0.html` 样式规范**，直接生成 v1.0 版本初步 PRD MD 格式草稿，初版 PRD 需搭建完整文档结构骨架（参考上面的标准文件夹结构），并深度填充以下核心内容：

1. **项目基本信息**（项目名称、版本、负责人、创建时间等）；

2. **需求背景与目标**：必须包含四列表格（目标类型、描述、衡量指标、目标值），明确需求价值和落地标准；

3. **用户使用场景与旅程图**：采用表格或清晰结构化表述用户旅程（User Journey Map），包含"阶段、用户触点、用户行为、痛点/情绪、产品机会点"五大核心要素，增强场景代入感；

4. **详细的功能清单与基础逻辑**：不可仅罗列模块名称，需详细梳理核心操作主线、前置条件、基础业务规则及数据流向，确保逻辑闭环。

*（后续章节如"详细方案（带原型模块）、整体流程图"可先留空，并标注"待原型确认后补充"，避免过早定论交互细节）*

### 3.2 用户确认 【等待用户确认】

向用户明确提问："这是产品的第一版详细架构和业务逻辑，您看整体方向和基础逻辑是否准确？请对**Markdown 格式 PRD 源文档**进行完善补充并确认；若无问题，我们将进入【原型设计】阶段，通过具体界面进一步梳理交互细节和潜在遗漏功能。"

**🛑 强制暂停：此处必须停止输出，等待用户回复同意后，方可进入步骤四！**

## 步骤四：产出高保真 **HTML 格式 原型文件** (核心验证阶段)

**这是本工作流的核心特色环节**。新手产品经理往往在看到具体界面后，才能发现业务逻辑中的漏洞（如缺失返回按钮、未考虑空状态、交互流程不顺畅等），原型验证可提前规避此类问题。

### 4.1 原型规范

1. **严格使用本地资源初始化**：**必须先复制模板框架的代码到项目根文件夹中**，仅使用`/template/v1.0/assets/`文件夹下的本地资源文件完成初版原型HTML的生成工作

2. **严格使用模板**：**必须参考 `/template/v1.0/prototype_html/admin_prototype_v1.0.html`（后台端/PC Web）和 `/template/v1.0/prototype_html/app_prototype_v1.0.html`（移动端/Mobile Web）标准模板结构**，**默认参考 `/assets/vercel/DESIGN.md` 设计规范**，确保样式统一、专业。

3. **移动端/Mobile Web**：适配移动端交互习惯。通过固定尺寸手机模拟外框（包含状态栏、底部导航等视觉元素）内嵌iframe技术展示子页面，支持全局视图和单独视频 切换，并提供子页面切换按钮组。

4. **后台端/PC Web**：采用多级菜单，二次页面尽量使用弹窗展示；内嵌iframe技术展示子页面。

5. 必须包含核心交互状态（如默认页、弹窗展开、操作成功/失败提示、空状态等）；需要通过原生 JavaScript 或 URL Hash（`#page1`）实现页面切换，模拟真实操作流程。

6. **沙盒锁定支持 (Focus Mode)**：使用postMessage或URL hash技术实现功能点的联动切换和高亮展示，编写原生 JavaScript 时，加入 URL 参数解析逻辑（如 `?focus=feature_id`）；处于 Focus 模式时，使用FadeIn动画提示对应功能点，为后续 PRD 嵌入原型、实现"功能交互高亮展示"奠定基础。

### 4.2 💡 必须使用 DESIGN.md

默认使用SKILL提供的vercel 设计规范。

用户可以引入其他符合 Google DESIGN.md 规范的规范。如果本地未提供，可在 https://github.com/VoltAgent/awesome-design-md 搜索下载。

### 4.3 原型审查与 PRD 同步更新 (核心迭代循环) 【等待用户反馈】

- 当用户手工修改Markdown格式 PRD 源文档后，处理“**同步PRD**”指令，完成HTML格式 PRD 源文档和HTML格式原型的同步工作，实现PRD与原型同步。

- 将生成的**HTML格式 原型文件**代码或预览效果呈现给用户，引导用户全面审查。

- 启发式引导用户思考：“请查看这个界面，您觉得用户点击该按钮后，若出现网络中断应如何提示？当前界面的信息展示是否完整、清晰？”

- **🛑 强制暂停：此处必须停止输出，等待用户反馈修改意见！**

- 根据用户反馈，针对性修改原型界面及交互逻辑。

- **🚨 关键规则（PRD 实时同步）**：用户提出原型修改意见（如新增功能、调整交互逻辑、优化页面流转等）时，AI**必须同步修改**：① **Markdown格式 PRD 源文档**；② **HTML格式PRD文档**；③ **HTML格式 原型文件**。不仅要更新功能逻辑描述，还需确保原型引用路径与最新原型保持一致，实现PRD与原型双向同步。

- 仅当用户对**Markdown格式 PRD 源文档**明确表示“完全满意”“可以进入下一步”时，方可进入步骤五。

## 步骤五：输出流程图 (Mermaid)（可选）

原型验证通过后，业务逻辑已趋于清晰，此时绘制流程图，可更直观地呈现业务流转，便于团队协同和后续维护。

1. 采用**Mermaid**语法绘制（优先使用`flowchart TD`或`sequenceDiagram`，贴合业务流程展示需求）。

2. 绘制重点：聚焦用户核心操作主线，同时明确标注原型审查中发现的**异常分支**（如网络中断、空数据、权限不足等场景的流程处理）。

3. 语法规范：确保代码中无导致渲染错误的特殊字符（如未转义的中文括号），保证流程图可正常渲染展示。

---

## 步骤六：产出最终版 PRD 包 (联动版 PRD+原型)

整合前五个步骤的所有成果，输出最终详细产品需求文档包，更新根文件夹下 `index.html` 作为主控文档，实现版本切换、分栏展示、多产品形态切换及左右联动核心功能，确保交付物完整、可用。具体要求如下：

### 6.1 主控文档（根文件夹 index.html）和其他HTML文件核心要求

主控文档作为 PRD 与原型的统一入口，负责版本管理、分栏展示和联动控制，替代单版本 PRD 直接打开的方式。需通过 HTML+CSS+原生 JavaScript 实现，无外部依赖，核心代码样式**必须严格参考 `/template/index.html` 模板文件**：

#### 主控文档（保存路径：专属根文件夹/index.html）

> **🚨 强制指令**：严格使用 `/template/index.html` 模板结构和样式，仅替换项目名称、版本选项和文件路径。

#### PRD 文档功能标识设置代码参考（保存路径-v1.0示例：专属根文件夹/v1.0/prd_html/prd_v1.0.html）

> **🚨 强制指令**：严格使用 `/template/prd_html/prd_v1.0.html` 模板结构和样式，确保排版、色彩、表格样式一致。

#### 原型适配联动代码参考 - 后台端/PC Web（保存路径-v1.0示例：专属根文件夹/v1.0/prototype_html/admin_prototype_v1.0.html）

> **🚨 强制指令**：严格使用 `/template/v1.0/prototype_html/admin_prototype_v1.0.html` 模板结构和样式，确保 UI 风格统一。

#### 原型适配联动代码参考-App（保存路径-v1.0示例：专属根文件夹/v1.0/prototype_html/app_prototype_v1.0.html）

> **🚨 强制指令**：严格使用 `/template/v1.0/prototype_html/app_prototype_v1.0.html` 模板结构和样式，确保移动端/Mobile Web 原型规范一致。

说明：上述代码可直接复制使用，需根据实际项目名称、版本及功能标识调整对应内容；所有文件路径需严格遵循文件夹架构，确保联动功能正常生效。

1. **顶部区域（导航栏）**：


    - 左侧显示项目名称（与PRD项目信息保持一致），字体加粗、尺寸适中，确保清晰可辨。

    - 项目名称旁设置版本切换下拉菜单，支持选择所有已迭代版本（如v1.0、v1.1），点击后左侧PRD文档区、右侧原型区同步切换至对应版本，无页面刷新。

    - 右侧设置产品形态切换按钮组，支持快速切换后台端/PC Web、移动端/Mobile Web等不同形态的原型，适配多产品形态需求。

    - 样式要求：导航栏固定在页面顶部，不随滚动消失；背景色与PRD主色调统一，边框底部简洁，无冗余装饰，适配所有主流屏幕尺寸。

2. **主体区域（左右分栏）**：


    - 默认布局：左右分栏各占50%宽度，高度为页面剩余高度（扣除顶部导航栏），两侧均支持独立滚动，避免内容挤压、遮挡。

    - 分栏调整：支持拖拽分栏中间分隔线，调整左右区域宽度（限制范围：左侧最小30%、最大70%；右侧最小30%、最大70%）；调整后通过localStorage保存布局状态，刷新页面后保留调整结果。

    - 左侧区域（PRD文档区）：通过iframe嵌入当前选中版本的PRD文档（路径示例：`./v1.0/prd_html/prd_v1.0.html`），确保PRD原有排版、文件夹导航、流程图等功能正常展示，无样式错乱。

    - 右侧区域（原型区）：通过iframe嵌入当前选中版本的高保真原型（路径示例：`./v1.0/prototype_html/admin_prototype_v1.0.html` 或 `./v1.0/prototype_html/app_prototype_v1.0.html`），保留原型原有交互功能和Focus模式，适配后台端/PC Web、移动端/Mobile Web原型的正常展示。

3. **左右联动功能（核心）**：


    - 关联规则：为PRD文档中每个“详细功能逻辑”模块设置唯一标识（如`admin_feature_01`、`app_setting`），与原型中对应功能/界面模块的标识一一对应，确保联动无偏差。

    - 联动效果：① 自动联动：用户在左侧PRD文档区滚动，当某“详细功能逻辑”模块进入可视区域时，右侧原型区自动定位至对应模块，通过Focus模式高亮（如切换页面、隐藏其他模块、添加边框/阴影、轻微放大），并滚动至模块居中位置，无需手动操作；② 手动联动：每个“详细功能逻辑”模块添加“查看原型”按钮，点击按钮可直接触发右侧原型区对应模块的定位与高亮，提升操作效率。

    - 实现方式：通过原生JavaScript监听左侧iframe的滚动事件，解析当前可视区域的功能标识，通过postMessage向右侧iframe发送指令，触发原型定位和高亮；手动联动通过为“查看原型”按钮绑定点击事件，向主控文档发送功能标识指令，由主控文档转发至原型iframe，确保联动流畅无延迟。

### 6.2 最终版 PRD 文档（prd/ 文件夹下）规范

prd/文件夹下的各版本 PRD 文档（如 prd_v1.0.html），需配合主控文档实现联动功能，在原有结构基础上补充以下要求，整体采用 HTML 格式，确保可独立运行、正常查看。

1. **排版与 UI 规范**：

### 6.3 原型适配要求（prototype/ 文件夹下）

各版本原型需配合主控文档的联动功能，确保可快速定位、高亮对应功能模块，方便用户查看。**必须参考 `/template/v1.0/prototype_html/admin_prototype_v1.0.html`（后台端/PC Web）和 `/template/v1.0/prototype_html/app_prototype_v1.0.html`（移动端/Mobile Web）模板**，确保 UI 风格统一。可参考以下核心代码逻辑，确保定位和高亮正常，与原有规范保持一致：

```
// 原型接收主控文档联动指令的核心代码（嵌入 prototype 各版本 HTML 中）
// 用于接收主控文档发送的定位高亮指令，实现 PRD 与原型联动
window.addEventListener('message', function(e) {
    // 验证消息来源，避免跨域异常
    if (e.origin !== window.location.origin) return;
    const message = e.data;
    // 接收主控文档的高亮指令
    if (message.action === 'highlight' && message.featureId) {
        // 移除所有原型模块的高亮状态
        document.querySelectorAll('.prototype-module').forEach(module => {
            module.classList.remove('feature-highlight');
        });
        // 定位并高亮对应功能模块
        const targetModule = document.querySelector(`.prototype-module[data-prototype="${message.featureId}"]`);
        if (targetModule) {
            targetModule.classList.add('feature-highlight');
            // 滚动原型区至目标模块居中，提升体验
            targetModule.scrollIntoView({ behavior: 'smooth', block: 'center' });
        }
    }
});
```

代码建议：原型中所有功能模块需添加 `data-prototype` 属性，属性值与 PRD 文档中对应功能模块的 `data-feature` 属性值完全一致，确保联动无偏差；若原型为多页面，需在每个页面中嵌入上述代码，保证切换原型页面后联动功能正常。

1. 为每个功能/界面模块添加唯一标识，与PRD文档中对应功能的标识一致（如`admin_feature_01`），可通过给模块容器添加`id`或`data-feature`属性实现。

2. 完善 Focus 模式逻辑，新增接收主控文档指令的接口；收到定位高亮指令时，自动高亮对应模块（如添加 2px 红色边框、5px 阴影，持续 3 秒后恢复正常，或保持高亮至切换功能模块）。

3. 确保原型支持通过 postMessage 接收外部指令（如定位标识、切换 Focus 模式），兼容不同版本原型的联动逻辑，无跨域问题。

4. 后台端/PC Web、移动端/Mobile Web 原型的原有规范不变，需确保嵌入主控文档右侧区域后，界面完整、交互正常，适配分栏宽度调整。

## 版本迭代与管理 (Version Control)

当项目进入后续迭代阶段（如从`v1.0`升级至`v1.1`），需执行严格的结构化版本控制，确保历史版本可追溯、不被破坏，同时适配主控文档的版本切换功能，保障迭代流程规范。

建议用户先更新对应版本的 **Markdown格式 PRD 源文档**，确认后同步更新 **HTML格式PRD文档**和原型。

### 7.1 文件物理隔离与复制

- **绝对禁止直接覆盖历史版本**，避免历史数据丢失、无法回溯。

- 迭代`v1.1`版本时，复制 `v1.0` 到 `v1.1`，并将全部标有版本的文件名改为`v1.1`，先处理markdown版本prd文件中的版本标识和版本历史记录，确保与新版本一致，然后同步更新html版本prd文件。

- 所有迭代修改仅在`v1.1`版本文件中进行，保留`v1.0`版本作为历史快照，便于后续对比、回溯。

### 7.2 主控文档版本切换更新

- 迭代新版本后，需更新根文件夹`index.html`主控文档的版本切换下拉菜单，新增新版本选项（如v1.1），确保菜单选项与`./v1.1/prd_html/`、`./v1.1/prototype_html/`文件夹下的版本文件一一对应。

- 优化版本切换逻辑，切换至新版本时，左右分栏的iframe路径需同步更新（如切换至v1.1时，PRD路径为`./v1.1/prd_html/prd_v1.1.html`，prototype路径为`./v1.1/prototype_html/admin_prototype_v1.1.html`、`./v1.1/prototype_html/app_prototype_v1.1.html`），确保联动功能不受影响。

- 优化提示：若用户查看的是历史版本，可在主控文档顶部导航栏下方添加明显警告横幅（例：“您正在查看历史版本v1.0，点击此处前往最新版v1.1”），点击横幅可快速切换至最新版本，提升使用体验。

### 7.3 变更日志与双向联动

- 新版PRD的【版本记录】表格中，需详细记录本次迭代的具体变更（新增、修改、下线的功能点），标注对应功能标识，方便团队核对、关联查看。

- **PRD与原型强联动要求**：`prd_v1.1.html`中所有功能标识，需与`admin_prototype_v1.1.html`、`app_prototype_v1.1.html`中对应模块的标识完全一致；主控文档切换至v1.1版本后，左右联动功能需正常生效，无错乱、失效问题。

- 确保新版PRD和原型嵌入主控文档后，样式、交互、联动均正常，适配不同屏幕尺寸，无兼容性问题。

- **保存新的项目修改历史日志**，方便多Agent协同回溯，命名规范：agent标识+时间+版本+摘要（例：`./history/gpt_2026_03_27_v1.1_update.md`）。
